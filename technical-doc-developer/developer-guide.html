Developer guide

Cara aman mengubah, menambah fitur, dan mengetes project

Bagian ini berisi panduan kerja untuk developer. Tujuannya agar perubahan tetap selaras dengan arsitektur yang ada dan tidak merusak flow Excel automation yang sudah berjalan.

Prinsip perubahan

  1. Jangan taruh business logic di UI. UI cukup mengambil input, memanggil service, dan menampilkan hasil.
  2. Gunakan helper path safety. Untuk config/master/output runtime, pakai fungsi di app/utils/path_safety.py.
  3. Tambahkan validasi sebelum eksekusi. Jika menambah field YAML, update validator di config_service.py dan tambahkan test.
  4. Jaga compatibility cross-platform. Pakai pathlib atau os.path, jangan hardcode separator Windows/Linux.
  5. Pertahankan Excel master runtime. Master runtime saat ini memakai file Excel/CSV di masters/; jangan merekomendasikan migrasi master runtime ke CSV saja kecuali memang diminta.

Menambah step recipe baru

Urutan umum untuk menambah step baru di recipe engine:

  1. Tambahkan nama step ke SUPPORTED_RECIPE_STEP_TYPES di config_service.py.
  2. Tambahkan validasi field wajib step tersebut di validator recipe.
  3. Implementasikan function di recipe_service.py, idealnya dengan nama _apply_nama_step_step().
  4. Tambahkan cabang dispatch di execute_step_recipe().
  5. Buat test di tests/test_pipeline_service.py atau file test khusus recipe.
  6. Jalankan python -m ruff check . dan python -m pytest -q.

Menambah transform klasik

Transform klasik dikelola di transform_service.py dan divalidasi di config_service.py. Jangan hanya menambah implementasi tanpa validasi, karena config invalid bisa gagal terlambat saat user menekan execute.

# pola umum
SUPPORTED_TRANSFORM_TYPES.add("nama_transform")

def _apply_nama_transform(data_df, transform_cfg, log):
    # validasi kolom dan tipe data
    # return DataFrame baru atau copy yang dimodifikasi
    return data_df

Menambah output summary baru

Summary khusus berada di recipe_service.py. Tambahkan builder function yang menerima data_df dan options, mengembalikan DataFrame final. Kemudian hubungkan ke _build_summary_output_sheet(). Jika summary punya layout/styling khusus, pastikan output_service.py masih bisa menulisnya dengan layout plain atau standard.

Testing strategy

AreaTest yang disarankan
Path safetyTest path normal, Windows separator, absolute path, .., case-insensitive filename, dan output escape.
Config validatorTest valid payload dan invalid payload untuk field baru.
PipelineTest happy path minimal dengan source kecil, master kecil, config kecil, lalu baca workbook output.
Recipe stepTest step secara end-to-end melalui pipeline agar integrasi dengan YAML ikut tercakup.
UITest behavior yang bisa dipisah dari rendering berat, misalnya parsing periode dan state execute.

Debugging checklist

  • Jika source gagal dibaca, cek ekstensi, ukuran file, sheet name, dan apakah file Excel corrupt.
  • Jika job tidak muncul, cek enabled, validitas job_profiles.yaml, dan validitas config target.
  • Jika master lookup gagal, cek path master harus di bawah masters/, nama sheet, dan nama kolom key.
  • Jika output gagal, cek sheet name, kolom output yang diminta, dan apakah output path tetap di outputs/.
  • Jika UI freeze, pastikan proses berat tetap berjalan di worker thread, bukan langsung di main thread Tkinter.
Belajar konsep: test yang baik tidak hanya memastikan “kode tidak error”, tapi juga memastikan hasil output sesuai. Contohnya test pipeline membaca kembali workbook output dan mengecek kolom serta nilai summary.
