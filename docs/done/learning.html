Main Window Guide

Memahami app/ui/main_window.py dengan bahasa pemula

File ini adalah pusat UI desktop berbasis CustomTkinter. Tugas utamanya bukan memproses Excel secara langsung, tetapi menghubungkan klik user dengan service, memantau status, dan menampilkan hasil proses secara aman.

Peran file ini

Mengatur tampilan, state UI, interaksi user, preflight, worker background, log, dan status eksekusi.

Inti arsitektur

UI meminta aksi, service menjalankan logika bisnis, thread menangani kerja berat, dan queue mengirim hasil kembali ke UI.

Buka source terkait: app/ui/main_window.py, app/ui/components/__init__.py, app/services/__init__.py, app/utils/__init__.py.

Penjelasan per bagian file

1. Import dan konstanta

  • Import standar seperti Path, Thread, Queue, dan datetime dipakai untuk path file, kerja background, antrean event, dan log waktu.
  • import customtkinter as ctk berarti semua widget UI dibuat dari library CustomTkinter.
  • Import dari app.services berisi logika bisnis seperti mencari config, validasi source, preflight, dan menjalankan pipeline.
  • Import dari app.utils berisi utilitas seperti membuka file picker, membuka folder output, dan membersihkan pesan error.
  • PIPELINE_STEP_ORDER menyimpan urutan langkah pipeline supaya progress ditampilkan konsisten.

2. JobSettingsDialog

  • Class ini adalah popup untuk membuat, memilih, dan mengubah job.
  • class JobSettingsDialog(ctk.CTkToplevel) berarti class ini mewarisi jendela popup dari CustomTkinter.
  • __init__ menyiapkan state awal, ukuran window, variabel form, lalu memanggil _build_layout().
  • refresh() membaca ulang config dan job dari disk agar isi dialog tetap sinkron.
  • _save_job() menyerahkan penyimpanan ke service agar validasi tidak tersebar di UI.

Kenapa dipisah jadi dialog? Karena layar utama harus tetap fokus ke alur utama: pilih source, pilih job, lalu execute. Tugas sekunder seperti mengelola job lebih nyaman ditempatkan di popup.

3. DesktopApp

  • Ini adalah window utama aplikasi.
  • class DesktopApp(ctk.CTk) berarti class ini mewarisi root window utama.
  • Class ini menyimpan state UI seperti source aktif, job aktif, hasil preflight, thread worker, queue event, dan teks hint.
  • __init__() bertugas menyiapkan state, membangun layout, memuat job, dan mendaftarkan event window.

4. State UI

StringVar dipakai untuk menghubungkan nilai Python ke widget. Saat nilainya berubah, tampilan yang terhubung bisa ikut berubah otomatis.

5. Fondasi layout

_init_redesign_foundation_refs() menyiapkan atribut widget dengan nilai awal None agar aman dicek dari method lain saat UI dibangun bertahap.

6. Builder UI: _build_*

  • UI dipecah menjadi beberapa method kecil seperti _build_header_section(), _build_source_card(), dan _build_log_panel().
  • Pendekatan ini membuat file lebih mudah dipelajari daripada satu method panjang yang mencampur semua widget sekaligus.
  • _build_card_title() adalah helper untuk membuat pola judul kartu yang berulang.
Bagian Apa yang dikerjakan Kenapa penting
Source card Memilih dan membersihkan file input User harus tahu file mana yang aktif sebelum proses jalan
Job card Memilih pekerjaan dan melihat detail config Satu source bisa diproses dengan job yang berbeda
Execute card Menjalankan pipeline dan membuka folder hasil Titik keputusan utama ada di sini
Log panel Menampilkan jejak proses runtime User butuh observabilitas tanpa membuka terminal

7. Source, job, hint, dan status

  • _select_source() menangani file picker dan validasi awal source.
  • _apply_source_path() memusatkan efek samping setelah source berubah: update state, simpan sesi, log, dan trigger preflight.
  • refresh_jobs() membaca job profile dari folder config lalu hanya menawarkan job yang valid ke user.
  • _resolve_primary_hint() dan _resolve_execute_hint() menerjemahkan state internal menjadi kalimat panduan yang mudah dibaca.

8. Session state, log, preflight, dan execute

  • _persist_session_state() menyimpan source/job terakhir agar user bisa melanjutkan pekerjaan sebelumnya.
  • _append_log() menjadi satu pintu untuk semua log agar format timestamp dan sanitasi konsisten.
  • _schedule_preflight() menjalankan pemeriksaan otomatis saat source atau job berubah.
  • _execute_pipeline() adalah pintu masuk proses utama, tetapi hanya akan berjalan jika semua syarat sudah aman.

Kenapa preflight penting? Karena preflight mendeteksi masalah lebih awal, sehingga tombol Execute hanya aktif saat source, job, dan kondisi output sudah layak diproses.

9. Thread dan queue

  • _run_preflight_worker() dan _run_pipeline_worker() menjalankan kerja berat di thread background.
  • Hasilnya dikirim lewat Queue, lalu dibaca oleh _poll_preflight_events() dan _poll_worker_events().
  • Ini dipilih karena Tkinter tidak aman diubah langsung dari worker thread.

Masalah yang dihindari: jika kerja berat dijalankan langsung di main thread, window bisa terasa macet atau not responding.

10. Entry point

  • run_desktop_app(paths) adalah pintu masuk UI desktop.
  • Fungsi ini mengatur appearance/theme, membuat DesktopApp, lalu memanggil mainloop().
  • mainloop() adalah loop utama Tkinter; tanpa itu window tidak akan hidup dan responsif.

Walkthrough bertahap untuk pemula

Kalau kamu baru belajar, jangan baca file ini dari atas ke bawah secara buta. Lebih mudah kalau dibaca berdasarkan urutan peran di aplikasi.

  1. Mulai dari entry point. Cari run_desktop_app(paths). Dari sini kamu tahu titik awal UI dibuat dan tahu bahwa DesktopApp adalah class utama yang harus dipahami.
  2. Lanjut ke konstruktor DesktopApp.__init__. Di sini kamu melihat state apa saja yang disiapkan: source, job, thread, queue, status, hint, dan log. Ini membantu memahami “data apa yang hidup” selama aplikasi berjalan.
  3. Baca semua method _build_*. Fokus dulu ke struktur layar, bukan logika bisnis. Pahami bahwa panel kiri mengurus input dan konteks, sedangkan panel kanan fokus ke execute dan log.
  4. Pahami interaksi user paling awal. Baca _select_source() lalu _apply_source_path(). Dari sini terlihat bagaimana satu aksi user bisa memicu validasi, log, simpan sesi, dan preflight.
  5. Pahami pemilihan job. Baca refresh_jobs(), _selected_job(), dan _update_job_info(). Tiga method ini menjelaskan bagaimana dropdown diisi dan bagaimana UI tahu job mana yang benar-benar valid.
  6. Masuk ke preflight. Baca _schedule_preflight(), _start_preflight_worker(), _run_preflight_worker(), dan _poll_preflight_events(). Ini bagian penting untuk belajar pola thread dan queue.
  7. Baru masuk ke execute. Baca _execute_pipeline() dan perhatikan guard clause-nya. Ini contoh bagus bagaimana Python dipakai untuk “menolak lebih awal” kondisi yang belum aman.
  8. Tutup dengan worker utama dan log. Baca _run_pipeline_worker(), _poll_worker_events(), dan _append_log(). Di titik ini kamu akan melihat bagaimana hasil pipeline kembali ke UI.

Yang sebaiknya dicari saat membaca

  • Method ini mengubah state apa?
  • Method ini dipanggil oleh siapa?
  • Method ini hanya UI atau juga logika bisnis?
  • Apa efek sampingnya ke log, tombol, atau preflight?

Kesalahan umum pemula

  • Mengira semua method harus dibaca berurutan dari atas.
  • Mencampur peran UI dengan peran service.
  • Tidak membedakan main thread dan worker thread.
  • Melupakan bahwa StringVar dipakai untuk binding ke widget.

Peta alur aplikasi

Berikut alur aplikasi dari start sampai proses Execute berjalan dan hasilnya kembali ke UI.

run_desktop_app(paths)
-> set appearance/theme
-> buat DesktopApp(paths)
   -> __init__()
      -> siapkan state
      -> _build_layout()
         -> _build_header_section()
         -> _build_source_card()
         -> _build_job_card()
         -> _build_execute_card()

Alur saat user memakai aplikasi

  • User membuka aplikasi dan refresh_jobs() memuat daftar job valid.
  • User klik Pilih Source, lalu _select_source() membuka file picker.
  • Source divalidasi dengan validate_source_file(...).
  • Jika valid, _apply_source_path(...) menyimpan source dan memicu _schedule_preflight().
  • User memilih job di dropdown, lalu _on_job_selected() memicu preflight lagi.
  • _run_preflight_worker() menjalankan run_preflight(...) di thread background.
  • Hasil preflight masuk ke queue dan diproses oleh _poll_preflight_events().
  • Jika aman, tombol Execute aktif.
  • User klik Execute, lalu _execute_pipeline() membuat worker pipeline.
  • _run_pipeline_worker() menjalankan run_pipeline(...), mengirim log dan progress ke queue.
  • _poll_worker_events() membaca hasil, memperbarui status, dan menampilkan output/log.

Diagram visual alur source - preflight - execute - output

1. App Start

run_desktop_app() membuat DesktopApp, membangun layout, dan memuat daftar job awal.

2. User Pilih Source

_select_source() membuka file picker, lalu memvalidasi file yang dipilih.

3. Source Disimpan ke State

_apply_source_path() mengisi self.source_path, update label UI, simpan sesi, dan menulis log.

4. User Pilih Job

Dropdown job memilih JobProfileSummary valid yang sudah dibaca dari config.

5. Preflight Dijadwalkan

_schedule_preflight() memastikan source dan job lengkap, lalu menandai status Memeriksa....

6. Worker Preflight Jalan

_start_preflight_worker() dan _run_preflight_worker() menjalankan pemeriksaan di thread background.

7. Hasil Masuk ke Queue

Worker tidak menyentuh widget langsung; ia hanya mengirim event ke Queue.

8. Main Thread Memproses Hasil

_poll_preflight_events() membaca hasil queue lalu memperbarui status dan enable/disable tombol execute.

9. User Klik Execute

_execute_pipeline() memeriksa guard clause: source ada, job valid, preflight selesai, dan preflight mengizinkan eksekusi.

10. Worker Pipeline Jalan

_run_pipeline_worker() memanggil run_pipeline(...), lalu mengirim log, progress, sukses, atau error ke queue.

11. UI Menampilkan Hasil

_poll_worker_events() membaca queue, memperbarui log, status, dan path output terakhir.

12. User Lihat Output

User bisa membuka folder hasil lewat _open_outputs_dir() atau memulai sesi baru setelah proses selesai.

Referensi file terkait

app/ui/components/__init__.py Penanda area untuk widget reusable jika UI nanti dipecah menjadi komponen kecil.

Mental model sederhana

UI

Menerima klik, menampilkan status, dan mengelola widget.

Service

Mengerjakan logika bisnis seperti validasi, preflight, dan pipeline.

Thread

Menjalankan pekerjaan berat tanpa membuat UI macet.

Queue

Jalur aman untuk mengirim hasil dari worker ke UI.

Cara membaca file ini sebagai pemula

  1. Mulai dari run_desktop_app() untuk tahu titik masuk aplikasi.
  2. Baca DesktopApp.__init__ untuk tahu state dan setup awal.
  3. Baca semua method _build_* untuk paham susunan layar.
  4. Baca handler utama: _select_source(), _on_job_selected(), _schedule_preflight(), dan _execute_pipeline().
  5. Tutup dengan worker dan polling event agar paham hubungan thread dan UI.
