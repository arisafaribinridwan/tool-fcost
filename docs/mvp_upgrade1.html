Roadmap Upgrade

MVP Upgrade 1

Interaktif, secure, dan UI/UX lebih baik untuk penggunaan harian user non-teknis.

Tujuan Utama

  1. Mengurangi trial-error sebelum klik Execute.
  2. Membuat proses lebih interaktif dan mudah dipahami.
  3. Menurunkan risiko keamanan dan kesalahan operasional.
  4. Meningkatkan kenyamanan UI/UX tanpa arsitektur berat.

Prioritas MVP-minded

P1 Wajib dulu - kurangi error dan naikkan rasa aman.

P2 Sangat disarankan - UX dan observabilitas.

P3 Nice-to-have - polish kualitas produk.

Gunakan panah kiri/kanan atau tombol Next/Prev.

Status Repo

Update Implementasi Saat Ini

Sudah Jalan

  • Selector Pekerjaan berbasis registry configs/job_profiles.yaml.
  • Path boundary hardening untuk path runtime configs/, masters/, dan output.
  • Sheet symptom sudah memakai rule table baru.
  • Lookup symptom sudah mendukung equals, contains, dan regex.

Guardrail Aktif

  • Rule symptom diproses dengan priority ascending dan tie-break urutan row.
  • Regex memakai semantik Python search.
  • Regex di-compile sekali per rule.
  • Regex invalid dan pattern terlalu panjang memblokir execute.

Belum aktif: Preflight Check, Dry Run, dan Start New Session.

P1 - Wajib

1) Preflight Check Interaktif

Problem: user sering baru tahu masalah saat pipeline sudah jalan.

Solusi: tombol Preflight Check untuk validasi sebelum eksekusi.

Apa yang dicek

  • Source ada, extension valid, bisa dibaca.
  • YAML valid, master referensi tersedia.
  • Kolom wajib source/master ditemukan.
  • Konflik sheet output dan potensi overwrite.

Output ke User

Status: Ready, Warning, Blocked.

Severity: ERROR, WARNING, INFO.

Dampak: runtime error menurun, trust meningkat.

P1 - Wajib

2) Dry Run / Preview Ringkas

Problem: user butuh konfirmasi resep sebelum menulis output final.

Solusi: mode Dry Run (checkbox atau tombol).

  • Pipeline jalan sampai transformasi.
  • Tidak menulis output final ke outputs/ (atau file preview sementara).
  • Tampilkan preview: jumlah baris input vs output, daftar sheet, contoh 10 baris utama.

Dampak: mencegah output salah format dan hemat waktu user.

P1 - Wajib

3) Path Security Boundary

Problem: tool berbasis path rentan traversal dan symlink edge case.

Solusi: perketat boundary akses file di semua titik I/O.

  • Semua path source/config/master harus resolve di root yang diizinkan.
  • Tolak .., path absolut keluar root, dan symlink berbahaya.
  • Validasi extension secara eksplisit.
  • Jika invalid: beri error jelas dan blokir eksekusi.

Dampak: risiko akses file tidak semestinya berkurang.

P1 - Wajib

4) Log Sanitization

Problem: log bisa memuat path detail user atau nilai sensitif.

Solusi: sanitasi log sebelum ditampilkan/disimpan.

  • Mask user-home path yang terlalu detail jika tidak perlu.
  • Mask nilai sensitif: NIK, nomor rekening, email.
  • Hindari dump dataframe mentah ke log.

Dampak: lebih aman saat log dibagikan/screenshot.

P1 - Wajib

5) Guardrail Resource

Problem: file besar atau rusak bikin UI terasa hang.

Solusi: size limit + row limit + timeout tahap kritis.

  • Batas ukuran source default 50-100 MB (configurable).
  • Batas jumlah baris untuk mode interaktif.
  • Timeout untuk baca/transform tertentu.
  • Jika terlewati: status warning/blocked dengan instruksi jelas.

Dampak: aplikasi lebih stabil, ekspektasi user lebih terkelola.

P1 - Wajib

6) Start New Session

Problem: setelah run selesai, state lama masih menempel dan membingungkan.

Solusi: tombol Start New Session aktif setelah worker selesai.

State yang di-reset

  • source kosong, log kosong, status kembali Idle.
  • Target output kembali placeholder awal.
  • Execute nonaktif sampai source baru dipilih.
  • Config di-refresh ulang sesuai startup.

Yang tidak di-reset

  • File output yang sudah dibuat.
  • Isi folder uploads/ dan outputs/.
  • File config/master di runtime folder.
P1 - Wajib

7) Pilih Pekerjaan sebagai Entry Point

Problem: user operasional tidak familiar dengan istilah teknis Config YAML.

Solusi: ganti/bungkus selector menjadi Pekerjaan.

  • User memilih jenis pekerjaan, bukan file mentah.
  • Setiap pekerjaan dipetakan ke config utama yang tepat.
  • Deskripsi singkat pekerjaan ditampilkan di UI.
  • Gunakan registry configs/job_profiles.yaml.

Status saat ini: selector Pekerjaan, refresh job, dan ringkasan config/master sudah aktif. Metadata registry yang sudah dipakai: id, label, config_file, enabled.

Dampak: mengurangi salah pilih config dan lebih ramah non-teknis.

P1 - Wajib

8) Lookup Symptom Berbasis Regex

Problem: variasi symptom comment di lapangan sangat banyak, penuh singkatan, typo, dan frasa yang mirip tetapi tidak identik.

Solusi: ubah master symptom menjadi rule table yang lebih maintainable dan tambahkan lookup berbasis regex.

Struktur Master Baru

  • Sheet symptom memakai kolom priority, part_name, match_type, pattern, symptom, notes.
  • Rule diproses dari priority terkecil ke terbesar.
  • Fallback per part_name tetap dimungkinkan.

Guardrail Minimum

  • Compile regex sekali per rule.
  • Regex invalid memblokir execute dengan pesan jelas.
  • Panjang pattern dibatasi.
  • Perilaku existing equals/contains tetap aman.

Status saat ini: rule table symptom dan regex lookup sudah aktif di engine, lengkap dengan ordering priority, tie-break urutan row, compile-once, invalid regex blocking, dan regression test untuk equals/contains.

Dampak: master symptom lebih fleksibel, lebih mudah dirawat, dan lebih cocok untuk variasi data operasional nyata.

P2 - Sangat Disarankan

9-13) UX dan Observabilitas

Progress + Error Actionable

  • Step indicator: load source, config, masters, transform, write output.
  • Status per langkah: running/success/failed.
  • Format error: apa masalah, dampak, langkah perbaikan.

Compatibility + Session

  • Preflight kecocokan source terhadap Pekerjaan.
  • Recent Files / Use last session.
  • Konfirmasi overwrite: Replace, Keep both, Cancel.
P3 - Nice to Have

14-16) Polish yang Meningkatkan Perceived Quality

  • Job Summary panel: source/config, durasi, jumlah sheet, warning/error, lokasi output.
  • Drag-and-drop source: mempercepat input tanpa buka dialog file.
  • Empty state dan hint ramah: CTA jelas saat belum ada pekerjaan valid atau preflight gagal.
Implementasi Bertahap

Rencana 4 Fase - Blocking First

Fase 1 - Fondasi Domain

  • Job selector Pekerjaan + registry job_profiles.
  • Path boundary hardening.
  • Restrukturisasi master symptom.
  • Regex-based symptom lookup + guardrail dasar regex.

Exit criteria: entry point domain stabil, boundary aman, dan sheet symptom baru terbaca valid.

Status: empat fondasi inti fase ini sudah tercapai di repo saat ini.

Fase 2 - Safety Sebelum Execute

  • Preflight Check.
  • Preflight compatibility source vs pekerjaan.
  • Error message standard.
  • Overwrite confirmation.
  • Dry Run.

Exit criteria: user tahu kondisi input sebelum execute penuh dan mismatch mayor tertangkap lebih awal.

Fase 3 - Stabilitas Operasional

  • Resource guardrail size/timeout.
  • Log sanitization.
  • Progress step indicator.
  • Start New Session.

Exit criteria: proses panjang lebih transparan, lebih aman, dan sesi berikutnya tidak membawa konteks lama.

Fase 4 - Efisiensi Harian

  • Last session restore.
  • Job summary panel.
  • Empty state dan hint ramah.
  • Drag-and-drop source file.

Exit criteria: UX harian lebih cepat, lebih matang, dan lebih sedikit membingungkan user operasional.
